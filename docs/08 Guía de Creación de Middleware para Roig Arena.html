Guía de Creación de Middleware para Roig Arena

Esta guía explica cómo crear middleware personalizado para proteger rutas y verificar permisos en la API.

¿Qué es un Middleware?

Un middleware es una capa que filtra las peticiones HTTP antes de que lleguen a los controladores. Actúa como un “guardián” que verifica condiciones y decide si permite o bloquea el acceso.

¿Por qué usar Middleware?

Sin Middleware (❌ Código repetido)

// En cada método del controlador
public function store(Request $request)
{
    if (!auth()->user()->isAdmin()) {
        return response()->json(['error' => 'No autorizado'], 403);
    }
    
    // Lógica del controlador...
}

public function update(Request $request, $id)
{
    if (!auth()->user()->isAdmin()) {
        return response()->json(['error' => 'No autorizado'], 403);
    }
    
    // Lógica del controlador...
}
// Código duplicado en cada método

Con Middleware (✅ Centralizado)

// Middleware aplicado a la ruta
Route::middleware(['auth:sanctum', 'admin'])->group(function () {
    Route::post('/admin/eventos', [EventoController::class, 'store']);
    Route::put('/admin/eventos/{id}', [EventoController::class, 'update']);
});

// Controladores limpios
public function store(Request $request)
{
    // Solo lógica del controlador
}

Ventajas:

  • ✅ Código centralizado
  • ✅ Reutilizable
  • ✅ Fácil de mantener
  • ✅ Controladores más limpios

Middleware del proyecto

IsAdmin Middleware

Propósito: Verificar que el usuario autenticado sea administrador.

Ubicación: app/Http/Middleware/IsAdmin.php

Código completo:

<?php

namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;
use Symfony\Component\HttpFoundation\Response;

class IsAdmin
{
    /**
     * Handle an incoming request.
     */
    public function handle(Request $request, Closure $next): Response
    {
        if (!$request->user() || !$request->user()->isAdmin()) {
            return response()->json([
                'error' => 'No tienes permisos de administrador',
            ], 403);
        }

        return $next($request);
    }
}

Explicación:

  1. Verifica que haya un usuario autenticado ($request->user())
  2. Verifica que el usuario sea admin (isAdmin())
  3. Si no cumple, retorna error 403 (Forbidden)
  4. Si cumple, permite continuar ($next($request))

Registrar el Middleware

Paso 1: Editar bootstrap/app.php

Añadir el alias del middleware:

<?php

use Illuminate\Foundation\Application;
use Illuminate\Foundation\Configuration\Exceptions;
use Illuminate\Foundation\Configuration\Middleware;

return Application::configure(basePath: dirname(__DIR__))
    ->withRouting(
        web: __DIR__.'/../routes/web.php',
        api: __DIR__.'/../routes/api.php',
        commands: __DIR__.'/../routes/console.php',
        health: '/up',
    )
    ->withMiddleware(function (Middleware $middleware) {
        $middleware->api(prepend: [
            \Laravel\Sanctum\Http\Middleware\EnsureFrontendRequestsAreStateful::class,
        ]);

        $middleware->alias([
            'verified' => \App\Http\Middleware\EnsureEmailIsVerified::class,
            'admin' => \App\Http\Middleware\IsAdmin::class, // ← Añadir esta línea
        ]);
    })
    ->withExceptions(function (Exceptions $exceptions) {
        //
    })->create();

Opción 1: En rutas individuales

});

// Limitar API a 60 peticiones por minuto

    // Rutas protegidas

Apéndice: Middleware en Laravel para Principiantes

¿Qué son los Middleware?

Los middleware son filtros HTTP que procesan las peticiones antes de que lleguen a los controladores. Actúan como “guardianes” que verifican condiciones y deciden si permiten o bloquean el acceso.

Analogía simple

Imagina un club nocturno:

  • Cliente = Petición HTTP
  • Portero = Middleware
  • Interior del club = Controlador

El portero (middleware) verifica:

  • ¿Tienes entrada? (autenticación)
  • ¿Eres mayor de edad? (permisos)
  • ¿Hay espacio? (throttling)

Si todo está bien, te deja pasar. Si no, te bloquea.

Estructura de un Middleware

Ubicación

app/Http/Middleware/
├── IsAdmin.php
├── CheckSubscription.php
└── ...

Anatomía básica

<?php

// 1. NAMESPACE
namespace App\Http\Middleware;

// 2. IMPORTS
use Closure;
use Illuminate\Http\Request;
use Symfony\Component\HttpFoundation\Response;

// 3. CLASE
class IsAdmin
{
    /**
     * 4. MÉTODO HANDLE - Procesa la petición
     */
    public function handle(Request $request, Closure $next): Response
    {
        // Verificar condición
        if (!$request->user() || !$request->user()->isAdmin()) {
            // Bloquear acceso
            return response()->json(['error' => 'Sin permisos'], 403);
        }

        // Permitir continuar
        return $next($request);
    }
}

Imports necesarios

1. Closure (obligatorio)

use Closure;

¿Qué es? Función anónima que representa el siguiente paso.
¿Para qué? Pasar la petición al siguiente middleware o controlador.

2. Request (obligatorio)

use Illuminate\Http\Request;

¿Qué es? Objeto de la petición HTTP.
¿Para qué? Acceder a datos de la petición (usuario, parámetros, etc.).

3. Response (obligatorio)

use Symfony\Component\HttpFoundation\Response;

¿Qué es? Tipo de retorno del método handle.
¿Para qué? Definir el tipo de respuesta.

Método handle()

El método handle() es el corazón del middleware:

public function handle(Request $request, Closure $next): Response
{
    // 1. Código ANTES de la petición
    // (verificaciones, logs, etc.)
    
    if (!$condicion) {
        // Bloquear acceso
        return response()->json(['error' => 'Mensaje'], 403);
    }
    
    // 2. Pasar al siguiente middleware/controlador
    $response = $next($request);
    
    // 3. Código DESPUÉS de la petición
    // (modificar respuesta, logs, etc.)
    
    return $response;
}

Parámetros

  • $request - Objeto de la petición HTTP
  • $next - Función que pasa al siguiente paso
  • Retorna Response - Respuesta HTTP

Flujo de ejecución

Petición HTTP
    ↓
Middleware 1 (antes)
    ↓
Middleware 2 (antes)
    ↓
Controlador
    ↓
Middleware 2 (después)
    ↓
Middleware 1 (después)
    ↓
Respuesta HTTP

Tipos de Middleware

1. Middleware de verificación (antes)

public function handle(Request $request, Closure $next): Response
{
    // Verificar ANTES de ejecutar el controlador
    if (!$request->user()) {
        return response()->json(['error' => 'No autenticado'], 401);
    }
    
    return $next($request);
}

2. Middleware de modificación (después)

public function handle(Request $request, Closure $next): Response
{
    // Ejecutar el controlador
    $response = $next($request);
    
    // Modificar DESPUÉS
    $response->header('X-Custom-Header', 'Valor');
    
    return $response;
}

3. Middleware mixto (antes y después)

public function handle(Request $request, Closure $next): Response
{
    // ANTES
    Log::info('Petición recibida', ['url' => $request->url()]);
    
    $response = $next($request);
    
    // DESPUÉS
    Log::info('Respuesta enviada', ['status' => $response->status()]);
    
    return $response;
}

Usuario autenticado

        return response()->json(['error' => 'No autenticado'], 401);

    return $next($request);
}

Parámetros de la ruta

{

    // Verificar algo con el ID
    
    return $next($request);
}

Datos del request

public function handle(Request $request, Closure $next): Response
{

    // Verificar algo con el email
    
    return $next($request);

Modificar la petición

Puedes modificar la petición antes de pasarla:

public function handle(Request $request, Closure $next): Response
{
    // Añadir datos a la petición
    $request->merge(['timestamp' => now()]);
    
    // Modificar datos
    $request->merge(['email' => strtolower($request->email)]);
    
    return $next($request);
}

Modificar la respuesta

Puedes modificar la respuesta después del controlador:

public function handle(Request $request, Closure $next): Response
{
    $response = $next($request);
    
    // Añadir headers
    $response->header('X-Custom-Header', 'Valor');
    
    // Modificar contenido (cuidado)
    $content = json_decode($response->getContent(), true);
    $content['timestamp'] = now()->toIso8601String();
    $response->setContent(json_encode($content));
    
    return $response;
}

Errores comunes

Error: “Class not found”

Causa: Falta el namespace o no está registrado.
Solución: Verificar namespace y registro en bootstrap/app.php.

Error: “Too few arguments”

Causa: Falta el parámetro $next.
Solución: Asegurar que el método tenga Request $request, Closure $next.

Error: “Call to a member function on null”

Causa: $request->user() es null.
Solución: Verificar que el usuario existe antes de acceder a sus propiedades.

use Closure;
use Illuminate\Http\Request;
use Symfony\Component\HttpFoundation\Response;

class MiMiddleware
{
    public function handle(Request $request, Closure $next): Response
    {
        // Verificación
        if (!$condicion) {
            return response()->json(['error' => 'Mensaje'], 403);
        }
        
        return $next($request);
    }
}

Crear

sail artisan make:middleware NombreMiddleware

Registrar

// bootstrap/app.php
$middleware->alias([
    'nombre' => \App\Http\Middleware\NombreMiddleware::class,
]);

Usar

// En rutas
Route::middleware('nombre')->group(function () {
    // Rutas
});

// En controlador
$this->middleware('nombre')->only(['metodo']);
