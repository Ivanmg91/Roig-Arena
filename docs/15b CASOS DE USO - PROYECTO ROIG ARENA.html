CASOS DE USO - PROYECTO ROIG ARENA

Basado en las rutas, controladores, resources y middleware implementados.

CASOS DE USO - USUARIO NO AUTENTICADO

CU-01: Registrarse en el sistema

  • Ruta: POST /api/register
  • Controlador: AuthController@register
  • Input: nombre, apellido, email, password, password_confirmation
  • Output: UserResource + token
  • Validaciones: email único, password min 8 caracteres

CU-02: Iniciar sesión

  • Ruta: POST /api/login
  • Controlador: AuthController@login
  • Input: email, password
  • Output: UserResource + token
  • Validaciones: credenciales correctas

CU-03: Ver listado de eventos futuros

  • Ruta: GET /api/eventos
  • Controlador: EventoController@index
  • Output: EventoResource collection
  • Filtro: Solo eventos con fecha >= hoy

CU-04: Ver detalle de un evento

  • Ruta: GET /api/eventos/{id}
  • Controlador: EventoController@show
  • Output: EventoResource (con sectores, precios, asientos disponibles)

CU-05: Ver listado de sectores activos

  • Ruta: GET /api/sectores
  • Controlador: SectorController@index
  • Output: SectorResource collection
  • Filtro: Solo sectores activos

CU-06: Ver asientos disponibles de un evento

  • Ruta: GET /api/eventos/{eventoId}/asientos
  • Controlador: AsientoController@porEvento
  • Output: AsientoResource collection (con disponibilidad)

CU-07: Ver asientos de un sector específico en un evento

  • Ruta: GET /api/eventos/{eventoId}/sectores/{sectorId}/asientos
  • Controlador: AsientoController@porSector
  • Output: AsientoResource collection (filtrado por sector)

CASOS DE USO - USUARIO AUTENTICADO

CU-08: Ver mi perfil

  • Ruta: GET /api/user
  • Middleware: auth:sanctum
  • Controlador: AuthController@user
  • Output: UserResource

CU-09: Cerrar sesión

  • Ruta: POST /api/logout
  • Middleware: auth:sanctum
  • Controlador: AuthController@logout
  • Acción: Elimina token actual

CU-10: Reservar un asiento (añadir al carrito)

  • Ruta: POST /api/reservas
  • Middleware: auth:sanctum
  • Controlador: ReservaController@store
  • Service: ReservaService@reservarAsiento
  • Input: evento_id, asiento_id
  • Output: ReservaResource
  • Validaciones:
    • Asiento disponible
    • Sector disponible para el evento
    • No duplicar reserva
  • Protección: lockForUpdate() (race condition)
  • Temporizador: 15 minutos

CU-11: Ver mis reservas activas (mi carrito)

  • Ruta: GET /api/reservas
  • Middleware: auth:sanctum
  • Controlador: ReservaController@index
  • Service: ReservaService@obtenerReservasActivas
  • Output: ReservaResource collection (con tiempo restante)
  • Filtro: Solo reservas no expiradas

CU-12: Cancelar una reserva (quitar del carrito)

  • Ruta: DELETE /api/reservas/{id}
  • Middleware: auth:sanctum
  • Controlador: ReservaController@destroy
  • Service: ReservaService@cancelarReserva
  • Validaciones:
    • Reserva pertenece al usuario
    • Reserva está bloqueada (no vendida)

CU-13: Confirmar compra (convertir reservas en entradas)

  • Ruta: POST /api/compras
  • Middleware: auth:sanctum
  • Controlador: CompraController@store
  • Service: CompraService@procesarCompra
  • Input: reservas[] (array de IDs)
  • Output: EntradaResource collection
  • Validaciones:
    • Reservas pertenecen al usuario
    • Reservas no expiradas
    • Precios existen
  • Acciones:
    • Cambia estado a “vendido”
    • Crea entradas con QR
    • Transacción DB

CU-14: Ver mis entradas compradas

  • Ruta: GET /api/entradas
  • Middleware: auth:sanctum
  • Controlador: EntradaController@index
  • Output: EntradaResource collection (con QR)

CU-15: Ver detalle de una entrada

  • Ruta: GET /api/entradas/{id}
  • Middleware: auth:sanctum
  • Controlador: EntradaController@show
  • Output: EntradaResource (información completa + QR)

CASOS DE USO - ADMINISTRADOR

CU-16: Crear un evento

  • Ruta: POST /api/admin/eventos
  • Middleware: auth:sanctum + admin
  • Controlador: EventoController@store
  • Input: nombre, descripcion_corta, descripcion_larga, poster_url, fecha, hora
  • Validaciones:
    • Fecha única (solo 1 evento por día)
    • Formato hora H:i

CU-17: Actualizar un evento

  • Ruta: PUT /api/admin/eventos/{id}
  • Middleware: auth:sanctum + admin
  • Controlador: EventoController@update
  • Input: campos a actualizar
  • Validaciones: fecha única (excepto el mismo evento)

CU-18: Eliminar un evento

  • Ruta: DELETE /api/admin/eventos/{id}
  • Middleware: auth:sanctum + admin
  • Controlador: EventoController@destroy
  • Validaciones: No tiene entradas vendidas
  • Acción: Soft delete

CU-19: Crear un sector

  • Ruta: POST /api/admin/sectores
  • Middleware: auth:sanctum + admin
  • Controlador: SectorController@store
  • Input: nombre, descripcion, activo

CU-20: Actualizar un sector

  • Ruta: PUT /api/admin/sectores/{id}
  • Middleware: auth:sanctum + admin
  • Controlador: SectorController@update
  • Input: campos a actualizar

CU-21: Eliminar un sector

  • Ruta: DELETE /api/admin/sectores/{id}
  • Middleware: auth:sanctum + admin
  • Controlador: SectorController@destroy
  • Validaciones: No tiene asientos asociados con reservas/entradas

CASOS DE USO - SISTEMA AUTOMÁTICO

CU-22: Liberar reservas expiradas

  • Comando: reservas:liberar
  • Programación: Cada minuto (cron)
  • Service: LiberarReservasService@liberarExpiradas
  • Acción:
    • Busca reservas con estado “bloqueado”
    • Filtra por reservado_hasta < now()
    • Elimina reservas expiradas
    • Registra en logs
  • Resultado: Asientos quedan disponibles nuevamente
