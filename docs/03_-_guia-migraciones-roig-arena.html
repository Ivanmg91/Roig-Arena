Guía para Crear las Migraciones del Proyecto Roig Arena

Esta guía te explica paso a paso cómo implementar la base de datos para el sistema de venta de entradas del Roig Arena en tu proyecto Laravel Sail.

¿Qué vamos a crear?

Un sistema completo de gestión de entradas que incluye:

  • Gestión de usuarios (clientes y administradores)
  • Infraestructura física del estadio (sectores y asientos)
  • Eventos con precios dinámicos por sector
  • Sistema de reservas con temporizador
  • Ventas definitivas con código QR

Arquitectura de la base de datos

El diseño separa dos conceptos importantes:

  1. Infraestructura física - Los sectores y asientos del estadio (permanente)
  2. Lógica de eventos - Eventos, precios y disponibilidad (temporal)

Esto permite reutilizar los mismos asientos físicos para diferentes eventos con precios distintos.

Control de disponibilidad

El sistema implementa un control de disponibilidad en dos niveles:

  1. Nivel global (sectores.activo)

    • Permite activar/desactivar un sector completo del estadio
    • Útil para mantenimiento, remodelaciones o cierres temporales
    • Afecta a todos los eventos

  2. Nivel por evento (precios.disponible)

    • Permite activar/desactivar un sector para un evento específico
    • Útil para eventos con configuraciones especiales
    • Solo afecta al evento en cuestión

Ejemplo de uso:

  • Si sectores.activo = false → El sector no está disponible para ningún evento
  • Si sectores.activo = true y precios.disponible = false → El sector solo no está disponible para ese evento específico
  • Si sectores.activo = true y precios.disponible = true → El sector está disponible para ese evento

  • Tener Laravel Sail corriendo: ./vendor/bin/sail up -d
  • Estar en el directorio del proyecto: cd /home/mrojo/laravel/Proyecto 4/Sail/arena
  • Laravel Sanctum ya instalado y configurado ✅
  • Modelo User ya actualizado con HasApiTokens y SoftDeletes ✅

Estado actual del proyecto

Ya hemos completado:

  • ✅ Instalación de Laravel Sanctum
  • ✅ Publicación de migraciones de Sanctum (tabla personal_access_tokens)
  • ✅ Modelo User actualizado con campos: nombre, apellido, is_admin

Lo que falta:

Paso 1: Modificar la migración de usuarios

Aunque el modelo User ya está actualizado, necesitamos modificar la migración para que la base de datos tenga los campos correctos.

1.1. Ubicar el archivo

cd database/migrations
ls -la | grep users

Busca el archivo: 0001_01_01_000000_create_users_table.php

1.2. Editar el archivo

Abre el archivo y modifica SOLO la tabla users en la función up():

Schema::create('users', function (Blueprint $table) {
    $table->id();
    $table->string('nombre');           // Cambiado de 'name'
    $table->string('apellido');         // Añadido
    $table->string('email')->unique();
    $table->timestamp('email_verified_at')->nullable();
    $table->string('password');
    $table->boolean('is_admin')->default(false);  // Añadido
    $table->timestamps();
    $table->softDeletes();              // Añadido
});

Importante: Deja las tablas password_reset_tokens y sessions tal como están.

Cambios realizados:

  • name → nombre y apellido (separados)
  • is_admin - Campo booleano para identificar administradores
  • softDeletes() - Permite borrado lógico

Paso 2: Crear la migración del estadio

Ahora crearemos todas las tablas relacionadas con el estadio y los eventos.

2.1. Generar el archivo de migración

./vendor/bin/sail artisan make:migration create_roig_arena_tables

Este comando crea un archivo nuevo en database/migrations/ con un nombre como:
2026_02_07_193317_create_roig_arena_tables.php

2.2. Editar el archivo creado

Abre el archivo recién creado y reemplaza todo su contenido con:

<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    /**
     * Run the migrations.
     */
    public function up(): void
    {
        // ============================================
        // 1. INFRAESTRUCTURA FÍSICA DEL ESTADIO
        // ============================================
        
        // Tabla de sectores (zonas del estadio)
        Schema::create('sectores', function (Blueprint $table) {
            $table->id();
            $table->string('nombre'); // Ej: VIP, Grada Norte, Tribuna
            $table->text('descripcion')->nullable();
            $table->boolean('activo')->default(true); // Control global (true = operativo, false = desactivado para TODOS los eventos)
            $table->timestamps();
        });

        // Tabla de asientos (butacas físicas)
        Schema::create('asientos', function (Blueprint $table) {
            $table->id();
            $table->foreignId('sector_id')->constrained('sectores')->onDelete('cascade');
            $table->string('fila'); // Ej: A, B, C
            $table->integer('numero'); // Ej: 1, 2, 3
            $table->unique(['sector_id', 'fila', 'numero']); // Un asiento único por sector
            $table->timestamps();
        });

        // ============================================
        // 2. LÓGICA DE EVENTOS (TEMPORAL)
        // ============================================
        
        // Tabla de eventos
        Schema::create('eventos', function (Blueprint $table) {
            $table->id();
            $table->string('nombre');
            $table->string('descripcion_corta', 255); // Para listados
            $table->text('descripcion_larga'); // Para la página del evento
            $table->string('poster_url')->nullable(); // URL de la imagen
            $table->date('fecha')->unique(); // Solo un evento por día
            $table->time('hora')->nullable(); // Hora del evento
            $table->timestamps();
            $table->softDeletes(); // Borrado lógico
        });

        // Tabla de precios (precio por sector en cada evento)
        Schema::create('precios', function (Blueprint $table) {
            $table->id();
            $table->foreignId('evento_id')->constrained('eventos')->onDelete('cascade');
            $table->foreignId('sector_id')->constrained('sectores')->onDelete('cascade');
            $table->decimal('precio', 10, 2); // Precio con 2 decimales
            $table->boolean('disponible')->default(true); // Control por evento (true = disponible, false = cerrado para este evento)
            $table->unique(['evento_id', 'sector_id']); // Un precio por sector/evento
            $table->timestamps();
        });

        // Tabla de estado de asientos (control de reservas)
        Schema::create('estado_asientos', function (Blueprint $table) {
            $table->id();
            $table->foreignId('evento_id')->constrained('eventos')->onDelete('cascade');
            $table->foreignId('asiento_id')->constrained('asientos')->onDelete('cascade');
            $table->foreignId('user_id')->nullable()->constrained('users')->onDelete('set null');
            $table->enum('estado', ['bloqueado', 'vendido']);
            $table->timestamp('reservado_hasta')->nullable(); // Temporizador de reserva
            $table->unique(['evento_id', 'asiento_id']); // Un estado por asiento/evento
            $table->timestamps();
        });

        // ============================================
        // 3. VENTAS DEFINITIVAS
        // ============================================
        
        // Tabla de entradas (ventas confirmadas)
        Schema::create('entradas', function (Blueprint $table) {
            $table->id();
            $table->foreignId('user_id')->constrained('users')->onDelete('cascade');
            $table->foreignId('evento_id')->constrained('eventos')->onDelete('cascade');
            $table->foreignId('asiento_id')->constrained('asientos')->onDelete('cascade');
            $table->decimal('precio_pagado', 10, 2); // Precio al momento de la compra
            $table->string('codigo_qr')->unique(); // Código QR único para validación
            $table->unique(['evento_id', 'asiento_id']); // Una entrada por asiento/evento
            $table->timestamps();
        });
    }

    /**
     * Reverse the migrations.
     */
    public function down(): void
    {
        // Eliminar en orden inverso por las dependencias
        Schema::dropIfExists('entradas');
        Schema::dropIfExists('estado_asientos');
        Schema::dropIfExists('precios');
        Schema::dropIfExists('eventos');
        Schema::dropIfExists('asientos');
        Schema::dropIfExists('sectores');
    }
};

Paso 3: Entender la estructura

Tabla: sectores

Define las zonas físicas del estadio.

Campos:

  • id - Identificador único
  • nombre - Nombre del sector (VIP, Grada, etc.)
  • activo - Control global (true = operativo, false = desactivado para TODOS los eventos)
  • descripcion - Descripción opcional
  • timestamps - created_at y updated_at

Ejemplo de datos:

id | nombre        | descripcion
1  | VIP           | Zona preferente con servicios premium
2  | Grada Norte   | Zona de animación
3  | Tribuna       | Zona cubierta

Tabla: asientos

Los asientos físicos del estadio.

Campos:

  • id - Identificador único
  • sector_id - A qué sector pertenece
  • fila - Letra de la fila (A, B, C…)
  • numero - Número del asiento
  • unique(['sector_id', 'fila', 'numero']) - Garantiza que no haya duplicados

Ejemplo de datos:

id | sector_id | fila | numero
1  | 1         | A    | 1
2  | 1         | A    | 2
3  | 1         | B    | 1

Tabla: eventos

Los eventos que se realizan en el estadio.

Campos:

  • id - Identificador único
  • nombre - Nombre del evento
  • descripcion_corta - Para mostrar en listados
  • descripcion_larga - Descripción completa
  • poster_url - URL de la imagen del evento
  • fecha - Fecha del evento (única)
  • hora - Hora del evento
  • softDeletes() - Permite cancelar eventos sin borrarlos

Ejemplo de datos:

id | nombre              | fecha      | hora
1  | Concierto Rock 2026 | 2026-03-15 | 20:00
2  | Final Copa          | 2026-04-20 | 18:30

Tabla: precios

Define el precio de cada sector para cada evento.

Campos:

  • id - Identificador único
  • evento_id - Qué evento
  • sector_id - Qué sector
  • disponible - Control de disponibilidad del sector para este evento (boolean)
  • precio - Precio en euros
  • unique(['evento_id', 'sector_id']) - Un precio por combinación

Ejemplo de datos:

id | evento_id | sector_id | precio
1  | 1         | 1         | 150.00  (VIP para concierto)
2  | 1         | 2         | 50.00   (Grada para concierto)
3  | 2         | 1         | 200.00  (VIP para final)

Tabla: estado_asientos

Controla qué asientos están reservados o vendidos para cada evento.

Campos:

  • id - Identificador único
  • evento_id - Para qué evento
  • asiento_id - Qué asiento
  • user_id - Quién lo tiene reservado/comprado
  • estado - ‘bloqueado’ (en carrito) o ‘vendido’ (pagado)
  • reservado_hasta - Timestamp de expiración de la reserva
  • unique(['evento_id', 'asiento_id']) - Un estado por asiento/evento

Ejemplo de datos:

id | evento_id | asiento_id | user_id | estado    | reservado_hasta
1  | 1         | 1          | 5       | bloqueado | 2026-02-07 20:00:00
2  | 1         | 2          | 3       | vendido   | null

Lógica:

  • Cuando un usuario añade al carrito → estado=‘bloqueado’, reservado_hasta=ahora+15min
  • Si paga antes de que expire → estado=‘vendido’, se crea entrada
  • Si expira → se elimina el registro, el asiento queda libre

Tabla: entradas

Las ventas confirmadas y pagadas.

Campos:

  • id - Identificador único
  • user_id - Quién compró
  • evento_id - Para qué evento
  • asiento_id - Qué asiento
  • precio_pagado - Precio al momento de compra (histórico)
  • codigo_qr - Código único para validar en la entrada
  • unique(['evento_id', 'asiento_id']) - Una entrada por asiento/evento

Ejemplo de datos:

id | user_id | evento_id | asiento_id | precio_pagado | codigo_qr
1  | 3       | 1         | 2          | 150.00        | QR123ABC456
2  | 5       | 1         | 5          | 50.00         | QR789XYZ012

Solución de problemas

Error: “SQLSTATE[42S01]: Base table or view already exists”

Causa: Las tablas ya existen.

Solución:

# Revertir todas las migraciones
./vendor/bin/sail artisan migrate:rollback

# O resetear completamente
./vendor/bin/sail artisan migrate:fresh

Error: “SQLSTATE[HY000] [2002] Connection refused”

Causa: MySQL no está corriendo.

Solución:

./vendor/bin/sail up -d
./vendor/bin/sail ps

Error en la sintaxis de la migración

Causa: Error de código en el archivo PHP.

Solución:

  1. Revisa el archivo de migración
  2. Verifica que no falten comas, puntos y comas, o paréntesis
  3. Ejecuta: ./vendor/bin/sail artisan migrate

Quiero empezar de cero

# Eliminar todas las tablas y volver a crearlas
./vendor/bin/sail artisan migrate:fresh

# Eliminar tablas y ejecutar seeders (cuando los tengas)
./vendor/bin/sail artisan migrate:fresh --seed

Comandos útiles

# Ver el estado de las migraciones
./vendor/bin/sail artisan migrate:status

# Revertir la última migración
./vendor/bin/sail artisan migrate:rollback

# Revertir todas las migraciones
./vendor/bin/sail artisan migrate:reset

# Resetear y ejecutar todas las migraciones
./vendor/bin/sail artisan migrate:fresh

# Ver las consultas SQL que se ejecutarán (sin ejecutar)
./vendor/bin/sail artisan migrate --pretend

Integración con Sanctum

Tabla personal_access_tokens

Esta tabla es creada automáticamente por Sanctum y almacena los tokens de autenticación:

Campos principales:

  • tokenable_type y tokenable_id - Relación polimórfica con User

  • abilities - Permisos del token (JSON)
  • last_used_at - Última vez que se usó

Relación con users:

// Un usuario puede tener múltiples tokens
User::find(1)->tokens; // Todos los tokens del usuario

// Crear un token
$user->createToken('auth-token')->plainTextToken;

// Eliminar tokens
$user->tokens()->delete(); // Todos
$user->currentAccessToken()->delete(); // Solo el actual

APENDICE

Estas notas te ayudarán a entender qué son las migraciones, cómo funcionan y el vocabulario específico que usa Laravel para trabajar con bases de datos.

¿Qué son las migraciones?

Las migraciones son como control de versiones para tu base de datos. Son archivos PHP que describen cómo crear, modificar o eliminar tablas y columnas.

Analogía simple

Imagina que estás construyendo una casa:

  • Planos arquitectónicos = Migraciones
  • La casa construida = Base de datos
  • Modificaciones al plano = Nuevas migraciones

Si trabajas en equipo, todos pueden tener los mismos “planos” (migraciones) y construir la misma “casa” (base de datos) en sus computadoras.

¿Por qué usar migraciones?

Sin migraciones (poblemático)

-- Cada desarrollador ejecuta SQL manualmente
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(255)
);

-- ¿Quién ejecutó qué? ¿Cuándo? ¿En qué orden?
-- Difícil de sincronizar entre equipos

Con migraciones (o rganizado)

// Archivo versionado en Git
// Todos ejecutan: php artisan migrate
Schema::create('users', function (Blueprint $table) {
    $table->id();
    $table->string('name');
});

Ventajas:

  • ✅ Historial de cambios
  • ✅ Reversibles (rollback)
  • ✅ Compartibles con el equipo
  • ✅ Automáticas en producción
  • ✅ Independientes del motor de BD (MySQL, PostgreSQL, etc.)

Estructura de un archivo de migración

Ubicación

database/migrations/
├── 2026_02_07_214245_create_users_table.php
├── 2026_02_07_214300_create_posts_table.php
└── 2026_02_08_091500_add_phone_to_users_table.php

Nombre del archivo:

  • 2026_02_07_214245 - Timestamp (año_mes_día_hora_minuto_segundo)
  • create_users_table - Descripción de lo que hace
  • .php - Es un archivo PHP

El timestamp asegura que las migraciones se ejecuten en orden cronológico.

Anatomía de una migración

<?php

// 1. IMPORTS NECESARIOS
use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

// 2. CLASE ANÓNIMA QUE EXTIENDE MIGRATION
return new class extends Migration
{
    /**
     * 3. MÉTODO UP - Ejecuta los cambios
     */
    public function up(): void
    {
        Schema::create('users', function (Blueprint $table) {
            $table->id();
            $table->string('name');
            $table->timestamps();
        });
    }

    /**
     * 4. MÉTODO DOWN - Revierte los cambios
     */
    public function down(): void
    {
        Schema::dropIfExists('users');
    }
};

Imports necesarios

1. Migration

use Illuminate\Database\Migrations\Migration;

¿Qué es? La clase base que todas las migraciones deben extender.
¿Para qué? Proporciona la estructura básica (métodos up/down).

2. Blueprint

use Illuminate\Database\Schema\Blueprint;

¿Qué es? El “constructor” de tablas. Define columnas y restricciones.
¿Para qué? Es el objeto $table que usas dentro de las funciones.

3. Schema

use Illuminate\Support\Facades\Schema;

¿Qué es? La fachada que interactúa con la base de datos.
¿Para qué? Crear, modificar o eliminar tablas.

Estos 3 imports son OBLIGATORIOS en toda migración.

Métodos principales

up() - Aplicar cambios

public function up(): void
{
    // Aquí defines QUÉ crear o modificar
    Schema::create('posts', function (Blueprint $table) {
        $table->id();
        $table->string('title');
    });
}

Se ejecuta con: php artisan migrate

down() - Revertir cambios

public function down(): void
{
    // Aquí defines CÓMO deshacer los cambios
    Schema::dropIfExists('posts');
}

Se ejecuta con: php artisan migrate:rollback

Regla de oro: down() debe deshacer exactamente lo que hace up().

Schema - El gestor de tablas

Schema es la clase que manipula la estructura de la base de datos.

Crear una tabla

Schema::create('nombre_tabla', function (Blueprint $table) {
    // Definir columnas aquí
});

Modificar una tabla existente

Schema::table('nombre_tabla', function (Blueprint $table) {
    // Añadir o modificar columnas
});

Eliminar una tabla

Schema::dropIfExists('nombre_tabla');

Renombrar una tabla

Schema::rename('nombre_viejo', 'nombre_nuevo');

Verificar si existe

if (Schema::hasTable('users')) {
    // La tabla existe
}

if (Schema::hasColumn('users', 'email')) {
    // La columna existe
}

Blueprint - El constructor de columnas

Blueprint es el objeto $table que usas para definir columnas.

Sintaxis básica

Schema::create('posts', function (Blueprint $table) {
    // $table es un objeto Blueprint
    $table->tipoDeDato('nombre_columna')->modificadores();
});

Glosario de tipos de datos

Tipos numéricos

id()

$table->id();

SQL equivalente: BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
Uso: Clave primaria autoincremental. Siempre la primera columna.

integer()

$table->integer('edad');

SQL: INT
Uso: Números enteros (-2,147,483,648 a 2,147,483,647)

bigInteger()

$table->bigInteger('visitas');

SQL: BIGINT
Uso: Números enteros muy grandes

tinyInteger()

$table->tinyInteger('nivel');

SQL: TINYINT
Uso: Números pequeños (0 a 255 o -128 a 127)

decimal()

$table->decimal('precio', 10, 2);

SQL: DECIMAL(10,2)
Uso: Números con decimales exactos (10 dígitos totales, 2 decimales)
Ejemplo: 12345678.90

float() / double()

$table->float('temperatura');
$table->double('coordenada');

SQL: FLOAT / DOUBLE
Uso: Números con decimales aproximados

Tipos de texto

string()

$table->string('nombre');          // VARCHAR(255)
$table->string('codigo', 10);      // VARCHAR(10)

SQL: VARCHAR(longitud)
Uso: Texto corto (hasta 255 caracteres por defecto)

text()

$table->text('descripcion');

SQL: TEXT
Uso: Texto largo (hasta 65,535 caracteres)

longText()

$table->longText('contenido');

SQL: LONGTEXT
Uso: Texto muy largo (hasta 4GB)

char()

$table->char('codigo_pais', 2);

SQL: CHAR(2)
Uso: Texto de longitud fija (ej: ‘ES’, ‘US’)

Tipos de fecha y hora

date()

$table->date('fecha_nacimiento');

SQL: DATE
Formato: YYYY-MM-DD
Ejemplo: 2026-02-08

time()

$table->time('hora_inicio');

SQL: TIME
Formato: HH:MM:SS
Ejemplo: 14:30:00

datetime()

$table->datetime('fecha_hora');

SQL: DATETIME
Formato: YYYY-MM-DD HH:MM:SS
Ejemplo: 2026-02-08 14:30:00

timestamp()

$table->timestamp('created_at');

SQL: TIMESTAMP
Uso: Fecha y hora con zona horaria

timestamps()

$table->timestamps();

Crea automáticamente:

  • created_at - Cuándo se creó el registro
  • updated_at - Cuándo se actualizó por última vez

Tipos booleanos

boolean()

$table->boolean('is_admin');

SQL: TINYINT(1)
Valores: true/false (1/0)
Uso: Campos de sí/no

Tipos especiales

enum()

$table->enum('estado', ['pendiente', 'aprobado', 'rechazado']);

SQL: ENUM('pendiente', 'aprobado', 'rechazado')
Uso: Campo con valores predefinidos

json()

$table->json('metadatos');

SQL: JSON
Uso: Almacenar datos en formato JSON

uuid()

$table->uuid('id');

SQL: CHAR(36)
Uso: Identificadores únicos universales

Modificadores de columnas

Los modificadores se añaden después del tipo de dato.

nullable()

$table->string('telefono')->nullable();

Significado: El campo puede ser NULL (vacío)
Sin nullable: El campo es obligatorio (NOT NULL)

default()

$table->boolean('activo')->default(true);
$table->integer('intentos')->default(0);
$table->string('idioma')->default('es');

Significado: Valor por defecto si no se especifica

unique()

$table->string('email')->unique();

Significado: No puede haber valores duplicados
SQL: Crea un índice UNIQUE

unsigned()

$table->integer('edad')->unsigned();

Significado: Solo números positivos (sin signo)

after()

$table->string('apellido')->after('nombre');

Significado: Coloca la columna después de otra específica

comment()

$table->string('codigo')->comment('Código único del producto');

Significado: Añade un comentario descriptivo a la columna

Foreign Keys (Claves foráneas)

Las foreign keys establecen relaciones entre tablas.

Sintaxis básica

$table->foreignId('user_id')->constrained('users');

Desglose:

  1. foreignId('user_id') - Crea columna BIGINT UNSIGNED
  2. constrained('users') - Crea la relación con tabla ‘users’
  3. Por defecto apunta a la columna ‘id’ de la tabla referenciada

Equivalente SQL

ALTER TABLE posts
ADD CONSTRAINT posts_user_id_foreign
FOREIGN KEY (user_id)
REFERENCES users(id);

Opciones de eliminación

onDelete(‘cascade’)

$table->foreignId('user_id')
      ->constrained('users')
      ->onDelete('cascade');

Significado: Si se elimina el usuario, se eliminan todos sus posts
Ejemplo: Borrar usuario → Borrar sus entradas

onDelete(‘set null’)

$table->foreignId('user_id')
      ->nullable()
      ->constrained('users')
      ->onDelete('set null');

Significado: Si se elimina el usuario, user_id se pone en NULL
Ejemplo: Borrar usuario → Sus posts quedan sin autor

onDelete(‘restrict’)

$table->foreignId('user_id')
      ->constrained('users')
      ->onDelete('restrict');

Significado: No permite eliminar el usuario si tiene posts
Ejemplo: Borrar usuario → ERROR si tiene posts

Opciones de actualización

onUpdate(‘cascade’)

$table->foreignId('user_id')
      ->constrained('users')
      ->onUpdate('cascade');

Significado: Si cambia el ID del usuario, se actualiza en posts

Índices

Los índices mejoran la velocidad de búsqueda.

index()

$table->string('email')->index();

Uso: Acelera búsquedas por este campo

unique()

$table->string('email')->unique();

Uso: Índice único (no duplicados)

Índices compuestos

$table->unique(['evento_id', 'asiento_id']);

Uso: La combinación de ambos campos debe ser única

primary()

$table->primary('id');

Uso: Define la clave primaria (raro, normalmente se usa id())

Soft Deletes (Borrado lógico)

softDeletes()

$table->softDeletes();

Crea: Columna deleted_at (TIMESTAMP NULL)
Significado: Los registros no se eliminan, solo se marcan como eliminados

Ejemplo:

// Eliminar (soft delete)
$user->delete(); // deleted_at = 2026-02-08 10:30:00

// El usuario sigue en la BD pero no aparece en consultas normales
User::all(); // No incluye usuarios eliminados

// Ver usuarios eliminados
User::withTrashed()->get();

// Restaurar
$user->restore(); // deleted_at = NULL

Crear una migración

Ejecutar migraciones

php artisan migrate

# Ver estado

# Ejecutar sin confirmación (producción)

Revertir migraciones

# Revertir último lote
php artisan migrate:rollback

# Revertir últimos 3 lotes
php artisan migrate:rollback --step=3

# Revertir todas

php artisan migrate:fresh

php artisan migrate:fresh --seed

Ver SQL sin ejecutar

php artisan migrate --pretend

Patrones comunes

Tabla de usuarios

Schema::create('users', function (Blueprint $table) {
    $table->id();
    $table->string('nombre');
    $table->string('email')->unique();
    $table->string('password');
    $table->boolean('is_admin')->default(false);
    $table->timestamps();
    $table->softDeletes();
});

Tabla con relación

Schema::create('posts', function (Blueprint $table) {
    $table->id();
    $table->foreignId('user_id')->constrained('users')->onDelete('cascade');
    $table->string('titulo');
    $table->text('contenido');
    $table->timestamps();
});

Tabla pivote (muchos a muchos)

Schema::create('post_tag', function (Blueprint $table) {
    $table->foreignId('post_id')->constrained('posts')->onDelete('cascade');
    $table->foreignId('tag_id')->constrained('tags')->onDelete('cascade');
    $table->primary(['post_id', 'tag_id']);
    $table->timestamps();
});

Añadir columna a tabla existente

Schema::table('users', function (Blueprint $table) {
    $table->string('telefono')->nullable()->after('email');
});

Eliminar columna

Schema::table('users', function (Blueprint $table) {
    $table->dropColumn('telefono');
});

Renombrar columna

Schema::table('users', function (Blueprint $table) {
    $table->renameColumn('name', 'nombre');
});

Modificar columna

Schema::table('users', function (Blueprint $table) {
    $table->string('email', 100)->change(); // Cambiar longitud
});

Buenas prácticas

1. Nombres descriptivos

// ✅ Bien
2026_02_08_create_eventos_table.php
2026_02_08_add_telefono_to_users_table.php

// ❌ Mal
2026_02_08_migration1.php
2026_02_08_fix.php

2. Una responsabilidad por migración

// ✅ Bien - Una migración por tabla
create_users_table.php
create_posts_table.php

// ❌ Mal - Todo en una migración
create_all_tables.php

3. Siempre implementar down()

// ✅ Bien
public function down(): void
{
    Schema::dropIfExists('posts');
}

// ❌ Mal
public function down(): void
{
    // Vacío
}

4. Usar constrained() con nombres explícitos

// ✅ Bien - Nombres en español
$table->foreignId('sector_id')->constrained('sectores');

// ❌ Puede fallar si la tabla no sigue convenciones inglesas
$table->foreignId('sector_id')->constrained();

5. Orden de las columnas

Schema::create('posts', function (Blueprint $table) {
    // 1. ID
    $table->id();
    
    // 2. Foreign keys
    $table->foreignId('user_id')->constrained('users');
    
    // 3. Campos principales
    $table->string('titulo');
    $table->text('contenido');
    
    // 4. Campos opcionales
    $table->string('imagen')->nullable();
    
    // 5. Timestamps
    $table->timestamps();
    
    // 6. Soft deletes
    $table->softDeletes();
});

Errores comunes y soluciones

Error: “Base table or view already exists”

Causa: La tabla ya existe en la base de datos.
Solución:

php artisan migrate:fresh  # Elimina todo y recrea

Error: “SQLSTATE[HY000]: General error: 1005”

Causa: Error en foreign key (tabla referenciada no existe).
Solución: Asegúrate de que las migraciones se ejecuten en orden correcto.

Error: “Class ‘Schema’ not found”

Causa: Falta el import.
Solución:

use Illuminate\Support\Facades\Schema;

Error: “Call to undefined method”

Causa: Método no existe en Blueprint.
Solución: Verifica la documentación de Laravel para tu versión.

{
    public function up(): void
    {

            // Columnas aquí
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('tabla');
    }
};

$table->id();                    // Clave primaria

// MODIFICADORES
->nullable()                     // Puede ser NULL

->unique()                       // Sin duplicados

      ->constrained('users')
      ->onDelete('cascade');
